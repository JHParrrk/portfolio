# 🚀 프론트엔드 리팩토링 아키텍처 개요 (Refactoring Architecture)

수정 전과 비교하여 변경된 핵심 아키텍처 및 기술 스택을 중요도 순으로 해설합니다.

## 🥇 1. 파일 및 폴더 구조의 변화 (FSD 기반 Feature-Driven Architecture)

### 📌 이전(Before)

- `src/components`, `src/commons` 등 코드가 **'역할(레이아웃, 버튼, 상태 등)'** 위주로 수평적으로 분리되어 있었습니다.
- **단점:** 특정 도메인(예: 게시판, 중고마켓)의 코드를 수정하려면 뷰, 로직, 유틸, 훅 폴더를 이리저리 돌아다녀야 하는 파편화 현상이 발생했습니다.

### ✨ 변경 후(After)

- **Feature-Sliced Design (FSD)** 철학을 바탕으로 **'도메인 기능(Feature)'** 위주로 디렉토리를 재구성했습니다.
- `src/features/`: `auth`, `boards`, `market`, `toast` 등 각 기능별로 독립적인 폴더를 가집니다. 그 안에 해당 비즈니스에만 쓰이는 `ui`, `api`, `models`를 응집시켜 놓았습니다.
- `src/shared/`: 여러 도메인에서 공통으로 쓰는 컴포넌트(Layout, Pagination 등)와 커스텀 훅 및 유틸리티를 모았습니다.
- `src/app/`: 전역 프로바이더(Apollo, 스타일 설정 등)와 같은 앱의 루트 설정을 묶었습니다.

### 💡 좋아진 점 (Benefit)

도메인별 코드 **응집도가 획기적으로 상승**했습니다. 게시판 관련 코드를 찾고 싶다면 고민 없이 `src/features/boards` 폴더만 열면 됩니다. 코드가 모듈화되어 프로젝트가 엔터프라이즈급으로 커져도 유지보수 비용이 급격히 줄어들며, 도메인 간 결합도를 낮출 수 있습니다.

## 🥈 2. CSS 및 스타일링 구조의 진화 (Zero-runtime CSS-in-JS)

### 📌 이전(Before)

- `styled-components` 또는 `Emotion`과 같은 **런타임 컴파일 CSS-in-JS**를 주로 혼용하거나 사용했습니다.
- **단점:** 브라우저(런타임)에서 JavaScript가 실행되면서 CSS를 생성&주입하므로 성능 오버헤드가 있었고, 무엇보다 Next.js 환경에서 첫 랜더링이나 페이지 이동(Link) 시 스타일이 풀리거나 깜빡이는(FOUC) 골치 아픈 이슈가 잦았습니다.

### ✨ 변경 후(After)

- `@vanilla-extract/css` 툴을 전면 도입했습니다.

### 💡 좋아진 점 (Benefit)

- **Zero-runtime:** 빌드 타임에 스타일이 정적 `.css` 파일로 완전히 추출됩니다. 런타임 환경에서 CSS를 계산할 필요가 없어 **초기 렌더링이 매우 빠르고 성능이 향상**됩니다.
- **안정성:** 브라우저에게는 일반 CSS 파일을 불러오는 것과 동일하게 작동하므로, Next.js 라우팅 시 스타일이 유실되거나 깜빡이지 않습니다.
- **Type-safe:** TypeScript 환경에서 완벽히 지원되어, 자동완성이 가능하며 오타로 인한 스타일 깨짐을 사전에(컴파일 단계에서) 방지할 수 있습니다.

## 🥉 3. 상태 관리 및 폼(Form) 핸들링 고도화

### 📌 이전(Before)

- 복잡한 컴포넌트의 폼(Form) 상태와 검증을 직접 `useState`나 무거운 로직으로 관리하여 보일러플레이트가 많고 컴포넌트 리렌더링이 자주 발생했습니다.

### ✨ 변경 후(After)

- **전역 상태:** `zustand` (토스트 팝업 등 UI 전역 상태용)
- **서버 상태:** `@apollo/client` (GraphQL 통신 및 서버 상태 캐싱)
- **폼 처리:** `react-hook-form` + `zod` (타입스크립트 스키마 기반 검증)

### 💡 좋아진 점 (Benefit)

- **`zustand` 전면 도입:** 무거운 기존 상태 관리 도구 대비 불필요한 보일러플레이트 코드가 없어져 세팅이 압도적으로 직관적이고 가벼워졌습니다.
- **`react-hook-form` + `zod`:** 가장 큰 변화 중 하나입니다. 입력할 때마다 리렌더링이 발생하는 문제를 `Uncontrolled Component` 기반으로 획기적으로 개선했으며, `yup`을 걷어내고 `zod`의 강력한 스키마 기반 유효성(Validation) 검사를 통해 TS 타입 시스템과 100% 호환되는 안정적인 폼 핸들링이 가능해졌습니다.

## 🏅 4. 마이크로 인터랙션 및 UX 인프라 (사용성 및 표준 준수)

### ✨ 변경 후(After)

- **헤드리스(Headless) UI 도입:** `@radix-ui/react-toast` 등 UI의 껍데기를 책임지는 Radix UI와 유연한 애니메이션 구현을 위한 `framer-motion` 패키지를 추가했습니다.
- **경로 설정 일원화:** 이중 혼용 상태이던 `jsconfig.json` 문제를 제거하고, `tsconfig.json`을 단일 진실 공급원으로 두어 `@/` 환경의 절대경로 매핑을 안정화시켰습니다.

### 💡 좋아진 점 (Benefit)

- 기존에는 디자인 요구사항에 맞추기 위해 접근성 구성을 무시하는 경우가 많았지만, Radix UI 등을 적용하여 키보드 네비게이션 및 스크린 리더(A11y)를 지원하는 웹 표준 친화적 서비스를 구성할 수 있는 기반이 마련되었습니다.
- 불필요하고 혼란스러웠던 터미널 및 빌드 에러 요인들이 정리되면서, 어지러운 상대 경로(`../../../../`) 지옥에서 벗어나 개발 생산성이 향상되었습니다.
